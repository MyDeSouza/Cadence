02 — Signal Classification

Four cognitive obligations

Signals are not sorted by urgency alone. They are sorted by what they demand from the user's working memory. Anything that cannot answer one of three questions clearly does not belong in the foreground.

Type A
Informational
"You have a meeting in 15 minutes."
Peripheral, low contrast, auto-dismissible. No user interaction required. Orientation only — updates situational awareness without entering the decision space.
Type B
Authorizational
"Meeting conflict detected. Proposed: reschedule to Friday 10am."
Central focus, high clarity. System has already resolved the complexity — user only confirms or declines. Single tap, single decision. This is the sweet spot of the triage engine.
Type C
Action-Bound
"Prepare slides for 2pm review."
Stable, persistent within a time window. Cannot be automated — user must act externally. Acknowledge or defer. Must not masquerade as a system-resolvable task.
Type D
Conflict Resolution
"Two meetings overlap at 1pm. Option A: Client review. Option B: Internal planning."
Highest cognitive load. Must be rare. Singular focus, no competing elements. Deliberate selection + confirmation. Justified by preventing future fragmentation.

04 — Interface Modes

Two surfaces, one engine

The same priority queue powers both interfaces. Mobile is reactive and foveal — what matters right now. Desktop is deliberate and spatial — the tools to do deep work.

Mobile — Signal View

The reactive surface. Foveal by design — surfaces only what requires a response right now. No calendar grid, no noise. One signal at a time, maximum 4 in the foreground.

Gesture-native: swipe to defer, tap to authorize, long-press to inspect the reasoning. Gestures terminate states — no dopamine loops, no infinite feeds.

Desktop — Tools View

The deliberate surface. Calendar-native with urgency layered in — colour and opacity encode priority, not just time. This is where deep work is orchestrated, not just scheduled.

05 — Build Order

What to tackle first

The sequence is designed so the engine works at every stage — even before any API integration is live.

01
Data model — normalized event object
Get the schema right before anything else. Every source feeds this object. This is the foundation everything else builds on.
DatabaseSchema
02
Manual input + scoring engine
The engine runs entirely on manually entered signals. No API dependency. Proves the core loop — ingest, score, surface.
Core LogicNo APIs needed
03
User preferences config
Source weights, keyword boosts, tag rules, quiet hours. The BOH layer. Lets the user tune the engine before any AI is involved.
BOHPhase 1 Rules
04
Google Calendar integration
First real external source. OAuth 2.0 flow, event normalization, real-time sync. Immediately proves the pipeline end-to-end.
Google APIOAuth 2.0
05
Output: daily digest
Simplest surface. Runs at user's set time, outputs top N items by score. Validates the scoring engine against real data.
Output LayerCron Job
06
Feedback logging — plant the seed early
Log every actioned / dismissed / ignored outcome. This is the training data for Phase 2. The AI layer needs this — build it now, use it later.
Learning LoopPhase 2 Prep
07
Email + app notification sources
Gmail API, Microsoft Graph for email. Webhooks for app notifications (Slack, etc.). Each new source normalizes into the same object.
Gmail APIWebhooks
08
AI scoring layer — Phase 2
Once feedback data exists, Claude API reads patterns — what gets actioned vs ignored, time-of-day behaviour, source reliability — and re-weights the rules dynamically. User always retains override.
Claude APIRequires Feedback Data
Recommended Stack
Backend
Node.js + Express or Python + FastAPI. Both have strong API client libraries for Google, Microsoft, and Anthropic.
Server
Database
PostgreSQL. Relational structure fits events, rules, and feedback logs cleanly. Add Redis later for priority queue caching.
Storage
Auth
OAuth 2.0 — required for Google and Microsoft API access. Handle this early; it unlocks most of the source integrations.
Security
AI Layer
Anthropic Claude API. Used for pattern reading, dynamic re-weighting, and natural language signal parsing in Phase 2.
Intelligence
Scheduling
Simple cron job to start — digest timing and score recalculation. Graduate to a proper queue (Bull/Redis) as volume grows.
Async
Mobile
React Native or Expo — single codebase, shared scoring logic with the web app. Gesture library for swipe/long-press interactions.
